§1Warum & Ziel ▶ Kurz: Die 3.0-Shell wird auf Dein Zielbild geschärft: klarere Navigation, ein rechtes Panel, echte Kontext-Sidebar, konsistente Farben, zwei Bugfixes.

Die Cursor-Shell 3.0 (Plan 002) steht, aber Dein Praxistest hat konkrete Lücken gezeigt: Das Terminal klemmt als Teilfenster unten rechts statt die Chat-Fläche zu ersetzen; die Activity Bar dupliziert mit Explorer-/Chat-/Terminal-Icons die Layout-Toggles; die Sidebar zeigt immer Dokumentenordner, egal wo Du bist; die Extensions-Ansicht ist noch ein „Store“ mit Unter-Tabs statt einer durchsuchbaren Liste; das Dashboard trägt eine KI-Modelle-Karte mit widersprüchlichen Zahlen (System-RAM vs. KI-RAM sind zwei verschiedene Messgrößen aus zwei Endpoints); und die Mitte hat einen anderen Hintergrund (--card) als die Panels (--background) — überall sichtbar.

Dazu kommen zwei echte Funktionsfehler: (a) Der Apps-Toggle schlägt bei Deinem Admin fehl — die Backend-Logs zeigen CSRF token missing für Deine PUT-Requests; Sessions von vor einem Deploy/Neustart verlieren ihren Token, und der Client beschafft ihn nicht neu. (b) „Datei → Neuer Ordner“ lässt Sidebar und Chat verschwinden — der Radix-Dialog setzt beim Öffnen aria-hidden auf Nachbar-Elemente, und genau darauf reagiert unsere Panel-Versteck-CSS-Regel.

Fertig heißt: Rechts eine Fläche mit [Chat | Terminal]-Umschalter (beide keep-alive), links nur Dashboard/Extensions/Apps, die Sidebar wechselt mit dem Kontext (Dokumente ⟷ Extension-Suche+Liste ⟷ zu), der Store ist eine Liste mit Mitte-Detail, das Dashboard zeigt nur Systemmetriken (Modell + KI-RAM korrekt in der Statusleiste), alle Flächen haben dieselbe Hintergrundfarbe, beide Bugs sind mit Regressionstests gefixt, die Workspace-Shell ist Standard — und alles ist per voller Matrix + Bug-Regressions-Flows auf dem Jetson belegt.

§2Scope ▶ Kurz: Shell-Feinschliff + zwei Bugfixes + Default-Flip; Chat-/RAG-Logik, Legacy-UI-Abbau und Backend-Schema bleiben unangetastet.

§3Architektur — einfach erklärt ▶ Kurz: Store v4 mit rightPanelMode, ein SidebarHost pro Kontext, Store als Liste+Detail, ein Hintergrund-Token — und zwei chirurgische Bugfixes.

Der Layout-Store bekommt Version 4: Statt zwei unabhängiger Booleans (chatVisible/terminalVisible) gibt es rightPanelVisible plus rightPanelMode („chat“ oder „terminal“). Ein neuer RightPanel-Wrapper hält beide Flächen dauerhaft gemountet (Keep-alive wie bisher — Streams und Terminal-Sessions überleben) und zeigt je nach Modus genau eine; der innere Split verschwindet. Vier bekannte Aufrufstellen außerhalb des Stores (Deep-Link-Brücken, „Terminal öffnen“, Chat-Scope) werden auf den Modus umgestellt, die Migration v3→v4 übersetzt bestehende Zustände verlustfrei.

Die Sidebar wird zum SidebarHost nach dem bewährten Brücken-Muster der Tabs: Aktiver Tab „Dashboard“ → Dokumente-Explorer (wie heute), „Extensions“ → Suchfeld + Liste aller Apps und Modelle (die Liste wird als gemeinsame Komponente promoted, damit die Feature-Isolationsregel hält), App-Tabs (n8n/Telegram/Datenbank) → Sidebar klappt zu. Der Store verliert seine Unter-Tab-Routen; die Mitte zeigt die Detailseite der links gewählten Extension (auf Basis der bestehenden Detail-, Aktivierungs- und Download-Bausteine); alte Deep-Links leiten um.

Farben: Die Mitte nutzt heute --card, alles andere --background — künftig gilt überall --background, Trennung nur per Border; ein Grep-Audit räumt alle abweichenden Flächen in den Workspace-Ansichten ab. Dashboard: Die KI-Karte entfällt; die Statusleiste zeigt aktives Modell + KI-RAM aus dem bereits vorhandenen /models/memory-budget-Endpoint (gemeinsamer React-Query-Cache, keine doppelte Poll-Last). Bugfixes: (a) Der API-Client beschafft einen fehlenden/abgelaufenen CSRF-Token automatisch neu und wiederholt den Request einmal — belegt ist die Ursache in den Backend-Logs (CSRF token missing bei Alt-Sessions); zusätzlich fällt der doppelte Fehler-Toast weg. (b) Die Panel-Versteck-Regel [data-panel][aria-hidden] kollidiert mit Radix-Dialogen, die aria-hidden auf Nachbarn setzen — die Sichtbarkeit bekommt ein eigenes Datenattribut (z. B. data-shell-hidden), das nur der Store setzt; die Hypothese wird vor dem Fix live im DOM verifiziert.

§5Akzeptanzkriterien ▶ Kurz: Vierzehn messbare Kriterien — jede Interview-Entscheidung und jeder Bug hat ein Abnahmekriterium.
  • Das rechte Panel hat oben einen Segment-Umschalter [Chat | Terminal]; der aktive Modus füllt die gesamte Fläche, es gibt kein unteres Terminal-Teilfenster mehr.
  • Beim Moduswechsel bleiben beide Seiten am Leben: laufender Chat-Stream und laufende Terminal-Session überleben den Wechsel; der letzte Modus wird nach Reload wiederhergestellt.
  • Die Activity Bar zeigt nur Dashboard, Extensions und aktivierte Apps — keine Explorer-/Chats-Icons oben, kein Terminal-Icon unten.
  • Oben rechts gibt es genau zwei Layout-Toggles (Sidebar, rechtes Panel) mit Tooltips.
  • Die Sidebar ist kontextabhängig: Dashboard → Dokumente/Projekte; Extensions → Suchfeld + Liste aller Apps und Modelle mit Status; n8n/Telegram/Datenbank → Sidebar zugeklappt.
  • Extensions haben keine Unter-Tabs mehr; Klick auf einen Sidebar-Eintrag öffnet die Detailseite in der Mitte mit funktionierendem Aktivieren/Installieren/Deinstallieren/Download; alte Store-Deep-Links leiten um.
  • Das Dashboard enthält keine KI-Modelle-Karte mehr; die Statusleiste zeigt aktives Modell + KI-RAM konsistent mit der tatsächlichen Ollama-Auslastung.
  • Alle Flächen (Sidebar, Mitte, rechtes Panel, Unterseiten) haben pixelgleich denselben Hintergrund; Trennung nur durch Borders — per Screenshot-Pixelvergleich belegt.
  • Apps-Toggle funktioniert auch in einer Alt-Session: fehlender CSRF-Token wird automatisch beschafft und der Request wiederholt; es erscheint höchstens EIN verständlicher Fehler-Toast, und nur wenn es wirklich scheitert.
  • „Datei → Neuer Ordner“ (und alle anderen Explorer-Dialoge/Menüaktionen) lassen Sidebar und rechtes Panel sichtbar; der Dialog funktioniert.
  • Die Workspace-Shell ist ohne localStorage-Flag der Standard nach Login; Opt-out und Legacy-Routen funktionieren weiter.
  • Store-Migration v3→v4 erhält Panel-Sichtbarkeit, Tabs und Terminal-Sessions (per Test mit echten v3-Daten belegt).
  • run-tests.sh --all und alle angepassten e2e-Specs sind grün.
  • Volle Playwright-Matrix + alle Bug-Regressions-Flows auf dem Jetson grün, 0 nicht-erklärte Konsolenfehler.
§6Verifikation auf dem Jetson ▶ Kurz: Wie bei Plan 002: Branch-Deploy mit Backup-Tag, volle Matrix — plus ein dedizierter Regressions-Flow pro gemeldetem Bug.

Vorgehen (vor dem Merge)

  1. Jetson: git tag -f backup-003 HEAD → Branch auschecken → docker compose up -d --build dashboard-backend dashboard-frontend → Health-Gate (Container healthy, HTTPS 200, Traefik-Warmup beachten).
  2. Vor jedem Playwright-Lauf: docker compose restart dashboard-backend (leert den Login-Rate-Limiter — bekannte Falle aus Plan 002).
  3. Temporärer Test-Admin nach dokumentierter Prozedur; nach dem Lauf löschen.

Volle Matrix + Bug-Regressionen (echter Browser gegen das Gerät)

  • Layout-Matrix: 4 Auflösungen (1024/1280/1440/1920) × Panel-Kombinationen (Sidebar auf/zu × rechtes Panel auf/zu × Modus Chat/Terminal) × Ansichten (Dashboard, Extensions inkl. Detail, n8n, Telegram) — Screenshots, kein horizontales Scrollen, Panel-Zustand == Toggle-Zustand.
  • Bug-Regression CSRF: Session einloggen → Backend neu starten (Token weg) → App-Toggle klicken → muss ohne Fehlermeldung durchgehen (Auto-Refetch + Retry); zusätzlich schneller Doppel-Toggle zweier Apps.
  • Bug-Regression Dialog: Datei → Neuer Ordner öffnen → Sidebar + rechtes Panel bleiben sichtbar → Ordner anlegen → erscheint im Explorer; dasselbe stichprobenartig für Upload/Rename/Delete und alle Menüpunkte.
  • Panel-Flow: Terminal-Befehl starten → auf Chat umschalten → Chat-Nachricht senden (Streaming) → zurück auf Terminal → Ausgabe komplett, genau eine WebSocket.
  • Kontext-Sidebar: Dashboard→Docs, Extensions→Suche (Suchbegriff filtert Liste), App-Tab→Sidebar zu, zurück→vorheriger Kontext.
  • Farb-Check: Pixelvergleich Hintergrund Sidebar vs. Mitte vs. rechtes Panel in Dashboard-, Extensions-, Telegram-Ansicht (muss identisch sein).
  • Statusleiste: angezeigtes Modell + KI-RAM gegen ollama ps auf dem Gerät geprüft.
  • Default-Flip: frischer Browser-Kontext ohne Flag → landet nach Login in der Workspace-Shell; Opt-out-Flag → Legacy erreichbar.
  • Konsole: 0 nicht-erklärte Fehler über alle Läufe (Whitelist nur für die dokumentierten vorbestehenden Meldungen).

Abschluss

Backend-/Traefik-Logs auf Fehler prüfen. Merge erst nach grüner Matrix; danach Gerät zurück auf main, Rebuild, Abschluss-Smoke. Gerät endet gesund auf main.
