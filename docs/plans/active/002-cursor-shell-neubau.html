§1Warum & Ziel ▶ Kurz: Terminal-Doppelung, starre Navigation und Größen-Wildwuchs verschwinden — die Shell wird konsequent Cursor.

Die heutige Workspace-Shell (Plan 001) ist funktional, aber logisch inkonsistent: Ein Klick auf „Terminal“ in der Activity Bar öffnet das Terminal als Tab in der Mitte, obwohl rechts bereits ein Terminal-Panel existiert — technisch sind das zwei unabhängig gemountete Instanzen derselben App, jede mit eigener WebSocket-Verbindung, die sich einen localStorage-Schlüssel teilen. n8n und Telegram-Bot stehen immer in der Navigation, auch wenn die Extension gar nicht aktiviert ist. Das Dashboard ist rechts abgeschnitten und reagiert nicht auf die Containerbreite, die Extensions-Seite wirkt um Klassen größer als der Rest, und unten links sitzt ein Chat-Toggle, der nur das rechte Panel ein-/ausklappt.

Dieser Plan baut die Shell als Ganzes neu — nach dem Vorbild von Cursor/VS Code: schmale Activity Bar, Sidebar, Mitte mit echter Tab-Leiste, rechtes Panel mit Chat und Terminal, Statusleiste, drei Layout-Toggles oben rechts. Ein zentraler Store verwaltet Panels, Tabs und Sessions; eine globale Kompakt-Skala normiert Typografie und Abstände in allen Ansichten.

Fertig heißt: Die Shell fühlt sich wie Cursor an — ein Raster, genau ein Terminal (im Panel, Sessions überleben das Ein-/Ausklappen), n8n/Telegram nur bei aktiver Extension, echte Tabs mit Restore, Dashboard und Extensions in kompakter, fluider Dichte — und die volle Playwright-Matrix (Ansichten × Panel-Kombinationen × 4 Auflösungen) ist auf dem Jetson im echten Browser grün, bei null Konsolenfehlern.

§3Architektur — einfach erklärt ▶ Kurz: Ein zentraler Store steuert Panels, Tabs und Sessions; das Terminal hat genau einen Mount; Sichtbarkeit kommt aus platform_apps.

Heute besitzt jede Oberfläche ihr eigenes Stück Layout-Wahrheit: Der Workspace-Store kennt „Explorer sichtbar“ und „LLM-Panel sichtbar“, das rechte Panel hat einen eigenen Chat/Terminal-Modus, und die Terminal-Ansicht führt zusätzlich eine eigene Tab-Liste im localStorage. Genau daraus entsteht der Doppel-Terminal-Bug: Mitte und Panel mounten unabhängig voneinander dieselbe Terminal-App — zwei WebSocket-Verbindungen, ein geteilter Speicher-Schlüssel.

Der Neubau ersetzt das durch einen zentralen Store (Version 3): Er verwaltet drei unabhängige Flächen (Sidebar, Terminal-Panel, Chat-Panel), die Tab-Leiste der Mitte und die Terminal-/Chat-Sessions. Das Terminal existiert nur noch als eine Instanz im rechten Panel; „ausblenden“ heißt verstecken statt zerstören (Keep-alive), damit laufende Prozesse weiterlaufen. Der Terminal-Klick in der Activity Bar toggelt nur noch dieses Panel — ein Terminal-Tab in der Mitte kann strukturell gar nicht mehr entstehen.

Die Sichtbarkeit von n8n/Telegram kommt aus der bestehenden platform_apps-Tabelle über useWorkspaceApps() — dieses Gating existiert bereits und wird konsequent auf alle Extension-Einträge angewandt statt sie hart zu verdrahten. Für die Dichte definiert das Design-System eine Kompakt-Skala (13-px-Basis, feste Abstandsstufen), an die Dashboard (auto-fit-Grid) und Extensions (Listenzeilen) angeschlossen werden.

§5Akzeptanzkriterien ▶ Kurz: Zwölf messbare Kriterien — von „genau eine Terminal-WebSocket“ bis „Matrix grün bei 0 Konsolenfehlern“.
  • Das Terminal erscheint ausschließlich im rechten Panel; der Terminal-Klick in der Activity Bar toggelt nur das Panel — ein Terminal-Tab in der Mitte existiert nicht mehr, auch nicht nach Reload.
  • Pro Terminal-Session besteht genau eine WebSocket-Verbindung (Dedup-Unit-Test + Netzwerk-Check im Browser).
  • Eine laufende Terminal-Session überlebt Panel-Toggle und Ansichtswechsel (laufender Prozess läuft sichtbar weiter).
  • n8n und Telegram erscheinen links nur bei installierter + aktivierter Extension; Deaktivieren entfernt den Eintrag ohne Reload, Aktivieren fügt ihn ohne Reload hinzu.
  • Der Chat-Toggle unten links existiert nicht mehr; drei Layout-Toggles oben rechts (Sidebar / Terminal / Chat) funktionieren unabhängig und zeigen Tooltips.
  • Tab-Leiste in der Mitte: mehrere Tabs gleichzeitig offen, schließbar, stabile Reihenfolge; nach Reload werden Tabs und aktiver Tab wiederhergestellt.
  • Das Dashboard passt sich jeder Containerbreite an: kein horizontales Scrollen, nichts abgeschnitten — in allen 4 Auflösungen × allen Panel-Kombinationen.
  • Die Extensions-Ansicht ist eine kompakte Liste (~32-px-Zeilen) mit Detailansicht per Klick.
  • Alle Ansichten nutzen die Kompakt-Skala (13-px-Basis, Token-Abstände); keine Ansicht wirkt sichtbar größer skaliert als der Rest.
  • Bestehende Nutzer verlieren beim Update weder offene Tabs noch Panel-Zustände (persist-Migration v2 → v3, per Test belegt).
  • Backend- und Frontend-Testsuiten (run-tests.sh --all) sowie die angepassten e2e-Specs sind grün.
  • Die volle Playwright-Matrix auf dem Jetson ist grün: alle E2E-Flows (Login, Chat senden, Terminal-Befehl, Extension-Toggle, Upload) funktionieren, 0 Konsolenfehler.
§6Verifikation auf dem Jetson ▶ Kurz: Branch-Deploy mit Backup-Tag, volle Playwright-Matrix im echten Browser — gemergt wird erst nach Grün.

Vorgehen (vor dem Merge)

  1. Auf dem Jetson (/home/arasul/arasul/arasul-jet): git tag -f backup-002 HEAD, dann Branch auschecken und docker compose up -d --build dashboard-backend dashboard-frontend.
  2. Health-Gate: alle Container healthy (docker compose ps), HTTPS 200 auf https://192.168.0.197/ bzw. https://100.121.244.80/ (Traefik braucht nach Frontend-Neustart ~30 s Warmup).
  3. Temporären Test-Admin nach dokumentierter Prozedur anlegen (bcrypt-Hash im Backend-Container, SQL per stdin) — wird nach dem Lauf wieder gelöscht.

Playwright-Matrix (im echten Browser gegen das Gerät)

  • Ansichten: Dashboard, Chat, Extensions, Einstellungen, Explorer, n8n (bei aktiver Extension), Terminal-Panel.
  • Panel-Kombinationen: Sidebar auf/zu × Chat auf/zu × Terminal auf/zu (8 Zustände).
  • Auflösungen: 1024×768, 1280×800, 1440×900, 1920×1080 — Screenshot je Zustand, Prüfung auf horizontales Scrollen/Abschneiden.
  • E2E-Flows: Login → Chat-Nachricht senden (Streaming-Antwort) → Terminal-Befehl ausführen → Panel toggeln (Prozess läuft weiter) → Extension deaktivieren/aktivieren (Activity-Bar-Eintrag verschwindet/erscheint) → Datei-Upload im Explorer → Reload (Tabs wiederhergestellt).
  • Konsole & Netzwerk: 0 Konsolenfehler über alle Läufe; genau eine Terminal-WebSocket-Verbindung.

Logs & Abschluss

docker compose logs dashboard-backend und traefik auf Fehler prüfen. Erst wenn Matrix und Health-Gate grün sind, wird der PR gemergt; danach Gerät zurück auf main, Rebuild und ein abschließender Smoke-Lauf.

Rollback

Vor dem Branch-Checkout auf dem Jetson wird backup-002 getaggt; schlägt Health-Gate oder Matrix fehl, geht das Gerät sofort per Checkout + Rebuild auf diesen Stand zurück — gemergt wird nur nach Grün. Nach dem Merge gilt: git revert des Merge-Commits + Rebuild. Eine DB-Migration ist voraussichtlich nicht nötig; falls doch (Nr. 101), ist sie additiv und idempotent, sodass ein Frontend-Rollback ohne Down-Migration gefahrlos bleibt. Der Layout-State im Browser migriert nur vorwärts (v3); bei einem Rollback startet die alte Shell mit Default-Layout — verloren gehen höchstens Fensterzustände, keine Daten.
