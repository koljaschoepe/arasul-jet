Was passiert
n8n bekommt einen festen Owner, der beim Start automatisch angelegt wird. Ein neuer Backend-Endpunkt meldet diesen Owner beim Öffnen des Tabs im Hintergrund an und setzt den n8n-Session-Cookie — der iframe lädt direkt den Editor.
Warum jetzt
Der Automationen-Tab zeigt heute eine fremde n8n-Sign-in-Wand und 401 auf /n8n/rest/login (Audit-Fund F2 aus Plan 006). Ein Kern-Feature der Appliance ist damit unbenutzbar.
Größtes Risiko
Der n8n-Session-Cookie ist Secure/SameSite-behaftet und könnte im iframe verworfen werden. Same-Origin sollte tragen, muss aber empirisch am Image 2.29.10 bestätigt werden.
Du entscheidest
  • Nahtlos statt Einmal-Login → fester Auto-Owner + Auto-Session
  • n8n auf dem Jetson wird komplett zurückgesetzt (keine echten Workflows vorhanden)
  • Scope strikt: nur die Anmelde-Wand, kein UI-Umbau

Offene Fragen an dich

Keine — alles Materielle wurde im Interview (10 Fragen, 3 Runden) entschieden. Der einzige noch offene Punkt ist rein technisch (exakter n8n-2.x-Cookie-Name / Same-Origin-Cookie-Weitergabe) und wird bei der Ausführung empirisch am Image 2.29.10 geklärt — keine Entscheidung deinerseits nötig.

§1Warum & Ziel ▶ Kurz: Der Automationen-Tab soll direkt den n8n-Editor zeigen — nie eine n8n-Anmeldung.

Arasul verspricht Automationen als Kern-Feature — im Dashboard unter „Automationen". Heute passiert dort das Gegenteil von plug-&-play: Der Tab bettet n8n ein, aber n8n zeigt seine eigene Sign-in-Maske und antwortet mit 401 auf /n8n/rest/login. Wer im Arasul-Dashboard bereits angemeldet ist, steht vor einer zweiten, fremden Anmeldung, die er gar nicht bedienen kann. Das Feature ist damit faktisch unbenutzbar (Audit-Fund F2 aus Plan 006).

Die Ursache ist strukturell: n8n läuft mit eigener Benutzerverwaltung (E-Mail/Passwort + JWT-Cookie). Traefik prüft per forward-auth zwar die Arasul-Anmeldung, meldet den Nutzer aber nicht bei n8n an — es gibt heute keinen Code-Pfad, der die n8n-Session herstellt. In n8n 2.29.10 lässt sich die Eigenanmeldung auch nicht einfach per Umgebungsvariable abschalten (2.x erzwingt einen Owner).

Die Lösung: ein einziger fester n8n-Owner, der beim Start automatisch angelegt wird, und ein schlanker Backend-Endpunkt, der diesen Owner beim Öffnen des Tabs transparent anmeldet und den n8n-Session-Cookie same-origin an den Browser gibt. Die Absicherung bleibt bei der Arasul-Anmeldung (Traefik-forward-auth) — n8ns eigene Wand wird schlicht überflüssig.

Fertig heißt: Ein im Dashboard angemeldeter Admin klickt auf „Automationen" und sieht sofort den n8n-Editor — ohne jemals eine n8n-Anmeldung zu sehen, ohne 401, mit voll funktionierender Workflow-Bearbeitung, auch nach einem Reload.

§2Scope ▶ Kurz: Nur die Anmelde-Wand wird behoben; kein UI-Redesign, keine neuen Features.

Im Scope

  • Auto-Login/401 beheben, sodass der Tab ohne sichtbare n8n-Anmeldung lädt
  • Fester n8n-Owner, idempotent beim Container-Start provisioniert
  • Neue Docker-Secrets n8n_owner_email / n8n_owner_password
  • Neuer Backend-Endpunkt GET /api/automations/session (stellt n8n-Session her)
  • Frontend: AutomationenTab holt die Session vor dem iframe
  • Einmaliger n8n-Reset auf dem Jetson (Volume + Schema) für den frischen Owner
  • Unit- + e2e-Test für den Session-Pfad; Doku-Sync (N8N.md, ENV-Vars)

Nicht im Scope

  • UI-Redesign des Tabs, Ausblenden der n8n-Navigation, Deep-Links
  • Neue Workflow-Features oder eigene Arasul-Automations-Oberfläche
  • Getrennte n8n-Konten pro Nutzer (bewusst Single-Tenant, ein Workspace)
  • n8n-Versions-Bump oder Downgrade
  • Zusätzliche n8n-interne Härtung (Public-API o. Ä.) — forward-auth genügt
  • Änderungen an Traefik-Routen, CSP oder X-Frame-Options
§3Architektur — einfach erklärt ▶ Kurz: Fester Owner + Backend-Session-Endpunkt melden n8n transparent an; forward-auth bleibt die Wand.

Heute ist der Automationen-Tab ein schlichter, gleicher-Herkunft-iframe mit src="/n8n/" (17 Zeilen, keine Auth-Logik). Traefik legt vor /n8n/ die forward-auth-Middleware, die beim Backend (/api/auth/verify) nur die Arasul-Session prüft und X-User-*-Header weiterreicht. Sie meldet dich aber nirgends bei n8n an. n8n selbst läuft mit eigener Benutzerverwaltung und liefert deshalb seine Login-Maske plus 401 auf /n8n/rest/login, solange kein gültiger n8n-Session-Cookie vorliegt. Diesen Cookie stellt heute niemand aus — das ist die ganze Bug-Ursache.

Nachher kommen zwei kleine Bausteine dazu, ohne die bestehende Absicherung anzufassen. Erstens legt der n8n-Container beim Start idempotent einen einzigen festen Owner an (Zugangsdaten aus zwei neuen Docker-Secrets, genau nach dem Muster des vorhandenen n8n_encryption_key). Zweitens ruft der Tab vor dem iframe einen neuen Backend-Endpunkt GET /api/automations/session auf; dieser meldet den festen Owner serverseitig bei n8n an (/rest/login) und gibt den n8n-Session-Cookie same-origin an den Browser weiter. Danach lädt der iframe /n8n/ bereits angemeldet — direkt im Editor.

Wichtig: Die Arasul-Anmeldung bleibt die einzige Wand. Wer nicht im Dashboard angemeldet ist, kommt schon an Traefik-forward-auth nicht vorbei — weder an den Editor noch an /api/automations/session. n8ns eigene Anmeldung entfällt für den Nutzer, ohne dass eine Schutzschicht verloren geht. Webhooks (/webhook/*) bleiben wie bisher bewusst öffentlich für externe Trigger.

§4Umsetzungsschritte ▶ Kurz: 7 Schritte: Secrets → Owner-Provisionierung → Backend-Route → Frontend → Tests → Doku → Jetson-Reset.

Schritt 1 — Owner-Secrets anlegen & verdrahten Risiko: niedrig

Zwei neue Docker-Secrets n8n_owner_email und n8n_owner_password nach dem exakten Muster von n8n_encryption_key: Deklaration in compose.secrets.yaml (file: ../config/secrets/<name>), Verdrahtung als *_FILE-Env in den n8n-Service. Idempotente Generierung im interaktiven Setup (neben den bestehenden n8n-Secrets). System läuft unverändert weiter — die Secrets sind vorhanden, aber noch ungenutzt.

compose/compose.secrets.yaml · scripts/interactive_setup.sh · config/secrets/ (generiert, nicht committet)

Ergebnis: Zwei Docker-Secrets n8n_owner_email/n8n_owner_password in compose.secrets.yaml deklariert und in n8n und dashboard-backend als *_FILE-Env verdrahtet; idempotente Generierung in arasul (setup_secrets) und interactive_setup.sh (Passwort n8n-policy-konform).

Schritt 2 — Idempotente Owner-Provisionierung im Entrypoint Risiko: mittel

Im services/n8n/entrypoint.sh (vor exec n8n) die Owner-Secrets per *_FILE-Shim auflösen, auf n8n-Readiness warten und — nur falls noch kein Owner existiert — den festen Owner über n8ns REST-Setup anlegen. Strikt idempotent: existiert der Owner bereits, passiert nichts. Kein Fehler bei erneutem Boot. Empirisch am Image 2.29.10 den korrekten Setup-Endpunkt/Ablauf bestätigen.

services/n8n/entrypoint.sh

Ergebnis: Entrypoint provisioniert den festen Owner idempotent: wartet auf n8n-Readiness (/rest/settings), legt ihn nur bei showSetupOnFirstLoad:true via POST /rest/owner/setup an (empirisch am Image 2.29.10 bestätigt: REST liegt an der Wurzel /rest/*, Owner-Body {email,firstName,lastName,password}). Läuft als Hintergrund-Job, stört den n8n-Start nie.

Schritt 3 — Backend-Route /api/automations/session Risiko: mittel

Neues Route-Modul routes/automations.js (Muster: routes/store/workflows.js): requireAuth + asyncHandler, kein try/catch. Meldet den festen Owner serverseitig bei n8n an (axios gegen N8N_HOST/N8N_PORT bzw. N8N_URL, wie tools/workflowsTool.js) und reicht den n8n-Set-Cookie attributgetreu (Secure/SameSite unverändert) same-origin an den Browser weiter. Bei n8n-Ausfall ServiceUnavailableError aus utils/errors.js. Registrierung in routes/index.js (Gruppe + router.use).

apps/dashboard-backend/src/routes/automations.js (neu) · apps/dashboard-backend/src/routes/index.js

Ergebnis: Neue Route routes/automations.js: requireAuth+asyncHandler, meldet den Owner über POST /rest/login ({emailOrLdapLoginId,password}) an und reicht den Set-Cookie (n8n-auth) attributgetreu durch; n8n-Ausfall → ServiceUnavailableError. Registriert in routes/index.js (+_meta).

Schritt 4 — Frontend: Session vor dem iframe holen Risiko: niedrig

AutomationenTab.tsx ruft beim Mount via useApi GET /api/automations/session auf (Muster: DocumentViewerTab.tsx — loading/error/cancelled-Guard), zeigt bis dahin einen schlanken Ladezustand und rendert den iframe erst, wenn der Cookie gesetzt ist. Bei Fehler eine klare Meldung statt der stummen Sign-in-Wand. TypeScript, Theme-Tokens, keine Hex-Literale.

apps/dashboard-frontend/src/features/workspace/viewers/AutomationenTab.tsx

Ergebnis: AutomationenTab.tsx holt beim Mount via useApi GET /automations/session (loading/error/cancelled-Guard), rendert den iframe erst nach Erfolg und zeigt bei Fehler eine klare Meldung statt der stummen Login-Wand. TS, Theme-Tokens.

Schritt 5 — Tests Risiko: niedrig

Backend-Unit-Test __tests__/unit/automations.test.js (Muster: workspaceApps.test.js, jest.mock('axios') + supertest): erfolgreicher Login reicht Set-Cookie durch; n8n unerreichbar → ServiceUnavailableError. Frontend-e2e: neuer Fall, der **/api/automations/session mockt und prüft, dass der iframe ohne Login-Maske lädt; bestehender e2e/workspace-shell.spec.ts bleibt grün.

apps/dashboard-backend/__tests__/unit/automations.test.js (neu) · apps/dashboard-frontend/e2e/*.spec.ts

Ergebnis: Backend-Unit-Test automations.test.js (4 Fälle: Cookie durchgereicht, n8n down→503, Login-ohne-Cookie→503, Auth nötig) — grün. Frontend-e2e-Fall in workspace-shell.spec.ts: mockt /api/automations/session+/n8n/, prüft iframe lädt ohne Login-Maske (läuft gegen den deployten Stack).

Schritt 6 — Doku synchronisieren Risiko: niedrig

docs/integrations/N8N.md §1 (Self-Service-Signup entfernen, festen Owner + Auto-Session beschreiben) und §8 (neue Secrets/Entrypoint-Schritt/Backend-Route ergänzen). docs/ENVIRONMENT_VARIABLES.md: veraltete N8N_BASIC_AUTH_USER/PASSWORD-„required"-Einträge korrigieren, neue Owner-Secrets + Secret→Env-Tabelle ergänzen.

Schritt 7 — Jetson: n8n zurücksetzen & Owner frisch provisionieren Risiko: mittel

Deploy-/Geräteschritt (in /work Phase 6–7): Vor dem Umschalten Volume arasul-n8n + Schema n8n sichern, dann leeren/neu anlegen, neues n8n-Image starten — der Entrypoint provisioniert dabei den frischen festen Owner. Nötig, weil ein evtl. halb-eingerichteter Owner die idempotente Provisionierung sonst überspringt. Anschließend Health prüfen und §6 fahren.

Gerät: docker compose -p arasul-platform (Volume arasul-n8n, Schema n8n) — kein committeter Code

§5Akzeptanzkriterien ▶ Kurz: Tab öffnet ohne Login, kein 401, Test-Workflow baubar, Session hält über Reload.
  • Der Automationen-Tab öffnet direkt den n8n-Editor — keine n8n-Sign-in-Maske ist sichtbar.
  • Kein 401 mehr auf /n8n/rest/login; im Netzwerk-Tab liegt eine gültige n8n-Session an.
  • 0 neue Console-Errors beim Öffnen des Tabs.
  • Ein Test-Workflow lässt sich anlegen, aktivieren und wieder löschen.
  • Nach einem Reload bleibt der Tab nahtlos angemeldet (Session hält).
  • Die Owner-Provisionierung ist idempotent: ein zweiter Container-Boot läuft ohne Fehler und ohne Duplikat.
  • Die Backend-Route liefert bei n8n-Ausfall einen sauberen ServiceUnavailableError (kein Crash, kein Leerlauf).
  • forward-auth bleibt wirksam: ohne Arasul-Anmeldung kein Zugriff auf /n8n oder /api/automations/session.
  • Alle bestehenden Tests bleiben grün (inkl. e2e/workspace-shell.spec.ts); neue Unit- + e2e-Tests decken den Session-Pfad ab.
§6Verifikation auf dem Jetson ▶ Kurz: Auf dem Jetson im echten Browser: Tab öffnen, Workflow anlegen→aktiv→löschen, 0 Console-Errors.

Verifikation läuft echt auf dem Jetson, nicht lokal. Reihenfolge:

Gerät & Deploy

  • n8n auf dem Gerät zurücksetzen (Schritt 7), neues Image ausrollen.
  • docker compose -p arasul-platform ps — n8n, dashboard-backend, traefik healthy.
  • Logs sichten: n8n-Entrypoint meldet „Owner provisioniert" (bzw. „bereits vorhanden"); Backend loggt GET /api/automations/session → 200; keine neuen Fehler.

Im echten Browser (Playwright-MCP gegen https://192.168.0.197/ bzw. Tailscale-IP)

  • Im Dashboard anmelden, „Automationen" öffnen → direkt der n8n-Editor, keine Login-Maske.
  • Netzwerk: /n8n/rest/login nicht mehr 401; Console: 0 Fehler.
  • Einen Test-Workflow anlegen → aktivieren → wieder löschen.
  • Seite neu laden → Tab bleibt nahtlos angemeldet.
  • Gegenprobe: in einem frischen Kontext ohne Arasul-Login /n8n/ aufrufen → von forward-auth abgewiesen.

Hinweis: Reachability über LAN (192.168.0.197) ist der zuverlässige Pfad; die Tailscale-SSH-Verbindung flackerte zuletzt (siehe Betriebs-Memo). Selbstsigniertes Zertifikat → ignoreHTTPSErrors ist in playwright.config.ts bereits gesetzt.
