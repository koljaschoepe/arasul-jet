§1Warum & Ziel ▶ Kurz: Aus dem Chat wird eine lokale Agenten-Orchestrierung — das eigentliche Kaufargument für Arasul.

Arasul verkauft sich als autonome Edge-KI auf eigener Hardware. Ein reines Chatfenster mit einem lokalen Modell ist dafür zu wenig — das kann jeder. Das unverwechselbare Versprechen ist: Prozesse automatisieren, indem man kleine lokale Modelle clever verkettet, jeweils mit dem richtigen Werkzeug und der richtigen Aufgabe ausgestattet, sodass am Ende ein Ergebnis herauskommt, für das andere ein großes Cloud-Modell bräuchten — vollständig DSGVO-konform auf dem Gerät.

Genau das baut dieser Plan: eine Agenten-Orchestrierungs-Schicht. Ein „Agent" ist eine gespeicherte Konfiguration aus Prompt, Modellwahl, erlaubten Tools und Rechten. Mehrere Agenten werden in einem visuellen Canvas zu einem Fluss verkettet — mit Verzweigungen (Wenn/Sonst) und parallelen Zweigen. Ein Fluss lässt sich per Slash-Command /flowname direkt aus dem Chat starten, per Zeitplan automatisch ausführen oder von n8n per HTTP anstoßen. Arasul wird damit das „KI-Gehirn", n8n bleibt eines der möglichen „Nervensysteme" für Trigger.

Wichtig für die Abgrenzung: Arasul baut keinen zweiten Workflow-Motor neben n8n. n8n kann Trigger, Scheduling und hunderte Konnektoren — das nutzen wir. Neu ist ausschließlich die KI-native Agenten-Schicht, die n8n nicht hat: Modell-Mischung lokal/Cloud, Tool-Rechte, verzweigte Agenten-Flüsse über eine GPU-Queue.

Fertig heißt: Ein Nutzer legt im Bereich „Agenten" mehrere Agenten an, verkettet sie im Canvas zu einem verzweigten Fluss, startet ihn per / im Chat, per Zeitplan oder aus n8n — und sieht das gestreamte Ergebnis, wobei lokale und Cloud-Modelle gemischt und die GPU nie überlastet wird.

§2Scope ▶ Kurz: Flow-Agenten (DB), Canvas, Provider-Registry, Tools, GPU-Queue, /-Chat, Scheduler + n8n-HTTP. Datei-Agenten und Audit-Log bleiben außen vor.

Im Scope

  • Flow-Agent-Datenmodell in PostgreSQL (Agenten, Flüsse, letzter Lauf)

§3Architektur — einfach erklärt ▶ Kurz: Eigener Bereich + Chat-Slash starten eine Fluss-Engine, die lokale/Cloud-Modelle über die GPU-Queue und lokale/externe Tools nutzt; Trigger via Scheduler oder n8n.

Heute spricht der rechte Chat direkt mit einem lokalen Modell. Künftig gibt es zwei Wege, einen Fluss zu starten — den neuen Agenten-Bereich mit Canvas und den Chat-Slash /flow — und beide münden in dieselbe Fluss-Engine im Backend. Diese Engine geht den gespeicherten Fluss-Graphen (JSON aus Knoten und Kanten) Schritt für Schritt durch, wertet Verzweigungen aus und führt parallele Zweige logisch gleichzeitig aus. Der entscheidende Trick für die Jetson-Hardware: jeder einzelne Modell-Aufruf läuft durch die bestehende llmQueueService-Warteschlange. Damit sind Zweige zwar „parallel", die GPU wird aber nie mit mehreren Modell-Läufen gleichzeitig überlastet.

Neu im Backend ist eine Provider-Registry: Der bisher fest auf Ollama verdrahtete Modell-Aufruf (toolLoop.js → callOllama) wird zu einer Abstraktion, die pro Agent entweder das lokale Ollama, einen OpenAI-kompatiblen Endpoint oder Anthropic anspricht. Die dafür nötigen API-Schlüssel liegen verschlüsselt in der Datenbank (gleiches AES-Muster wie die bestehenden Nutzer-Credentials) und sind nur für Admins editierbar. Agenten bauen und ausführen dürfen dagegen alle Nutzer.

Alles lebt sauber neben den bestehenden datei-basierten Workspace-Agenten (Plan 008): eigene DB-Tabellen, eigener Bereich, eigener Trigger / (das bestehende @ für Datei-Agenten bleibt unangetastet). Getriggert wird ein Fluss auf drei Wegen: manuell/Chat, über einen neuen Arasul-Scheduler (Cron im Backend) oder über den offenen Endpoint POST /api/agents/:id/run, den n8n per HTTP mit einem Bearer-Token aufruft — dasselbe Muster wie die bestehenden Sandbox-Run-Routen.

§5Akzeptanzkriterien ▶ Kurz: Konkret prüfbar — Agent bauen/ausführen, Modelle mischen, verzweigter Fluss, /-Chat, n8n-Trigger, Bestand grün.
  • Ein Flow-Agent (Prompt + Modell + Tools) lässt sich im Bereich „Agenten" anlegen, speichern und manuell ausführen; die Antwort streamt live.
  • Die Modellwahl pro Agent funktioniert zwischen lokalem Ollama und mindestens einem externen Provider (OpenAI-kompatibel/Anthropic); externe API-Keys sind AES-verschlüsselt gespeichert und nur für Admins editierbar.
  • Ein verzweigter Fluss (If/Else + mindestens ein paralleler Zweig) lässt sich im React-Flow-Canvas bauen, speichern und ausführen; parallele Zweige laufen logisch parallel, alle LLM-Aufrufe gehen durch die GPU-Queue.
  • Externe Tools (Web/HTTP) sind standardmäßig aus und nur nach Admin-Freigabe pro Agent nutzbar; ein rein lokaler Agent erreicht kein externes Netz.
  • /flowname im rechten Chat startet einen Flow-Agenten und streamt das Ergebnis; @agentname (Datei-Agenten, Plan 008) funktioniert unverändert weiter.
  • POST /api/agents/:id/run mit gültigem Bearer-Token liefert den Output (für n8n); ohne bzw. mit falschem Token → 401.
  • Ein Zeitplan (Cron) im Arasul-Backend löst einen Flow-Agenten automatisch aus; der letzte Lauf ist in der schlanken Lauf-Ansicht sichtbar.
  • Bestehende Tests (agentRun.routes, agentToolLoop, ActivityBar, AgentChatPanel) bleiben grün; neue Tests für Provider-Registry, /run-Route und Canvas-CRUD sind vorhanden.
  • Alle neuen Backend-Routen nutzen asyncHandler + custom errors; kein Route-Level-try/catch; API_ROUTE_GROUPS ist aktualisiert.
§6Verifikation auf dem Jetson ▶ Kurz: Gestuft im echten Browser: Einzelagent → Cloud-Lauf → verzweigter Fluss → /-Chat → Cron → n8n-HTTP.

Live im echten Browser gegen das Gerät (https://192.168.0.197), gestuft — jeder Schritt aus §4 wird nach seinem Deploy einzeln verifiziert, da es keinen Feature-Flag gibt:

  1. Erreichbarkeit & Bereich: Dashboard lädt, neuer Bereich „Agenten" erscheint in der Aktivitätsleiste.
  2. Einzelagent lokal: Agent mit Ollama-Modell anlegen → manuell ausführen → Antwort streamt live.
  3. Cloud-Lauf: Admin hinterlegt einen externen Key (verschlüsselt) → Agent mit externem Modell → ein Lauf liefert eine Antwort.
  4. Verzweigter Fluss: Im Canvas If/Else + einen parallelen Zweig bauen → ausführen → Knoten-Status live; in docker compose logs dashboard-backend ist zu sehen, dass die Modell-Aufrufe seriell durch die Queue laufen (keine GPU-OOM).
  5. Chat-Slash: /flowname startet den Fluss, Ergebnis erscheint im Chat; @agentname testweise weiterhin funktionsfähig.
  6. Zeitplan: kurzes Cron-Intervall setzen → ein Lauf entsteht automatisch, sichtbar in der Lauf-Ansicht.
  7. n8n-HTTP: n8n-HTTP-Node ruft POST /api/agents/:id/run mit Bearer-Token → 200 + Output; ohne Token → 401.

Health/Logs: docker compose ps (alle healthy), docker compose logs -f dashboard-backend (Queue-Reihenfolge, Provider-Dispatch, keine Fehler), GPU per tegrastats während eines Parallel-Flusses stabil.
