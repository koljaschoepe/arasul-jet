§1Warum & Ziel ▶ Kurz: Aus der fertigen Shell ein Werkzeug machen, das sich groß, konsistent und selbsterklärend anfühlt.

Die Workspace-Shell steht seit Plan 002/003 — Activity Bar, Tabs, rechtes Panel mit Chat⇄Terminal, Extensions als Liste, eine Flächenfarbe. Beim echten Arbeiten fällt aber auf: alles wirkt eine Spur zu klein (Dashboard, Dokumente-Explorer, die Reiter ganz links), einige Bereiche wirken unfertig oder unklar (Extensions-Detail, Terminal-Ordner, Datenbank-App), und Kleinigkeiten stören den nativen Eindruck (Chat-Platzhalter, fehlendes Anhang-Feedback, mittig statt links ausgerichtete Einstellungen).

Dieser Plan poliert das Frontend entlang des Kundennutzens: Der Kunde soll die Appliance öffnen und ohne Nachdenken loslegen können. Der rote Faden ist eine gemeinsame Komfort-Dichte (alles atmet gleich und groß genug) plus verify-first: Wir prüfen jeden Bereich zuerst live im Browser, weil mehrere gewünschte Dinge bereits existieren — dann schließen wir gezielt die echten Lücken statt Vorhandenes neu zu bauen.

Fertig heißt: Auf 1024–1920 px wirkt die ganze Oberfläche einheitlich groß und luftig, das Dashboard nutzt die volle Breite als Aktions-Hub, Extensions haben eine Übersicht mit reicher Detailseite und funktionierendem Download, der Chat zeigt angehängte Dateien sichtbar an, das Terminal macht klar in welchem Ordner man ist, die Einstellungen sind durchgängig linksbündig, und die Datenbank-App erklärt sich beim Öffnen selbst — alles live auf dem Jetson bestätigt.

§2Scope ▶ Kurz: Sieben Frontend-Bereiche polieren; Cloud-Sync, Mobil-Layout und Umbau kritischer Logik bleiben draußen.

Im Scope

  • Komfort-Dichte plattformweit (neue ui-Skala + alte Text-/Spacing-Skala auf ein gemeinsames, größeres Niveau gehoben)
  • Dashboard: volle Breite / responsive + Aktions-Hub-Kacheln (Chat, Upload, Projekt, zuletzt genutzt)
  • Extensions: Kategorie-Übersichtsseite + reiche Detailseite (inkl. Kontextlänge) + Download live verifiziert
  • Chat: sichtbare Anhang-Chips + Drop-Overlay + nativer Platzhalter — auf der Oberfläche, die der Kunde nutzt
  • Terminal: klarer Projekt-/Ordner-Header mit Wechsler
  • Einstellungen: durchgängig links-ausgerichtetes Raster in allen Panels
  • Datenbank-App: Befund (funktioniert die Logik?) + Onboarding/Klartext
  • Live-Verifikation auf 1024 / 1280 / 1440 / 1920 px

Nicht im Scope

  • GitHub-/OneDrive-Cloud-Sync — nur als neues Roadmap-Thema notiert
  • Mobil-/Handy-Layout unter 1024 px
  • Umbau der Chat-Sende-, Upload-, Auth- oder RAG-Logik (nur additive Backend-Felder erlaubt)
  • Neubau der Legacy-/chat-Route (nur Platzhalter/Feedback klären, kein Funktionsumbau)
  • Funktionsumbau der Datenbank-App ohne vorherigen Befund
§3Architektur — einfach erklärt ▶ Kurz: Eine gemeinsame Dichte-Skala für alle Screens + verify-first, um echte Lücken statt vermeintlicher zu füllen.

Der erste Hebel — eine Dichte, überall. Heute steuern die „ui"-Tokens (--text-ui-*, --spacing-ui-* in index.css) die Größe nur auf ~6 Dateien: Shell, Dashboard und Extensions. Der Rest der App — Einstellungen, Chat, Datenbank, Legacy-Routen — hängt an einer älteren, kleineren Skala. Deshalb wirkt manches kompakt und manches nicht: es sind zwei Systeme. Wir heben beide auf ein gemeinsames, komfortables Niveau an (Grundschrift, Zeilenhöhe, Abstände, Icon-Größen ~eine Stufe größer), zentral über die Token-Definitionen. So wird „größer" ein Schalter an einer Stelle — nicht 40 Handjustierungen.

Der zweite Hebel — erst schauen, dann bauen. Der Research-Durchlauf hat gezeigt: die Anhang-Chips + das Drop-Overlay im Chat, der Terminal-Projekt-Wechsler und die links-ausgerichtete Theme-Auswahl existieren bereits; auch die Modell-Detailseite ist schon reich an Infos. Der Kunde erlebt sie trotzdem als fehlend/unklar — also liegt das Problem woanders (falsche Oberfläche, echter Live-Bug, oder Auffindbarkeit). Jeder Schritt startet daher mit einem kurzen Live-Audit im echten Browser; erst der bestätigte Befund entscheidet, ob wir polieren, verdrahten oder eine echte Lücke (Kategorie-Übersicht, Kontextlänge, Aktions-Hub, Datenbank-Onboarding) neu bauen.

Was sich am System ändert. Fast alles bleibt im Frontend (CSS-Tokens + Komponenten). Genau eine kleine, additive Backend-Änderung ist vorgesehen: eine neue Spalte context_length im Modell-Katalog (Migration + Read-Feld), damit die Detailseite die Kontextlänge zeigen kann. Kritische Pfade (Chat-Senden, Auth, RAG) werden nicht angefasst.

Größter Live-Befund — die wahre „alles zu klein"-Ursache

Der globale Komfort-Hebel (S1, html { font-size: 106.25% }) griff bei 1920 px (17px) und <1280 px, aber nicht im Band 1280–1511 px — dort klemmte eine Legacy-Media-Query aus der alten Sidebar-UI die Root-Schrift hart auf 14.5px. Genau die typische Laptop-Breite (u. a. 1440er-Screens) → die eigentliche Ursache, warum alles zu klein wirkte, schon vor diesem Plan. Behoben in Folge-PR #206 (Merge d44f28a, Band auf 103.125% = 16.5px). Nach Redeploy live bestätigt: 1440 px → 16.5px (vorher 14.5px), ActivityBar-Button 41px (vorher 36), 0 Dashboard-Überläufer.
