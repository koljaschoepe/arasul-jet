# n8n Workflow-Engine — Findings

## Scope

n8n v1.121.1 (älter als 6 Wochen per Sentry-Log), 3 Custom-Nodes, 4 Pre-built Workflows

## BLOCKERS

Keine echten Blockers. n8n läuft stabil, ist aber unterintegriert.

## MAJORS

### N8N-M01: Kein SSO / Auto-Login mit Dashboard-Credentials

- n8n hat eigenes Login
- User muss separate Credentials merken
- Fix: Forward-Auth via Traefik + Custom-Middleware (`dashboard-backend /api/n8n/auth`)

### N8N-M02: Kein Frontend-Trigger-API

- Custom-Nodes vorhanden, aber Dashboard kann keine Workflows aus UI starten
- Chat-Feature könnte n8n-Flows triggern (z.B. "sende Mail an X"), aktuell nicht möglich
- Fix: `POST /api/n8n/trigger/:workflow_id` + Permission-Check

### N8N-M03: Workflows nicht versioniert in Git

- 4 Pre-built Workflows liegen im n8n-Volume, nicht im Repo
- Bei Neu-Setup neuer Jetson: Workflows fehlen oder sind out-of-date
- Fix: `n8n-workflows/*.json` als Code-Artefakte + Auto-Import beim Bootstrap

### N8N-M04: Custom-Nodes-Build nicht Teil von `./arasul bootstrap`

- 3 Custom-Nodes benötigen `npm install + link` manuell
- Fix: Build-Step im preconfigure.sh oder Dockerfile-Layer

### N8N-M05: n8n veraltet (Sentry Error-Tracking disabled)

- n8n 1.121.1 → aktuell 1.130+ (Stand 04/2026)
- Nicht dringend, aber Appliance sollte periodisch updaten

## MINORS

- N8N-m01: Keine n8n-Metriken im Metrics-Collector (Workflow-Fail-Rate, Execution-Time)
- N8N-m02: Kein Webhook-Inventory im Frontend
- N8N-m03: n8n-Credentials werden in n8n-eigener DB gespeichert, nicht in arasul_db
- N8N-m04: Kein Backup-Hook für n8n-Workflows (nur PostgreSQL-Dump, aber n8n nutzt eigene DB)

## OK / FUNKTIONIERT

- n8n Container `healthy`, startet zuverlässig
- 3 Custom-Nodes (Arasul-LLM, Arasul-RAG, Arasul-Telegram?) eingebunden
- Traefik-Routing für `/n8n/*` funktioniert
- Persistenz in Named-Volume `arasul-n8n`
- Workflows laufen (laut Snapshot)
